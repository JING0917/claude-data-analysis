# 数据质量监控计划

## 背景

对三张核心表建立数据质量监控体系，通过帆软BI报表展示 + 定时调度告警，确保数据质量可观测、可追溯、可报警。

### 监控表范围

| 表名 | 说明 | 核心场景 |
|------|------|---------|
| `dim.dim_silkworm_user` | 用户基础表 | 用户注册、画像、活跃分析 |
| `dwd.dwd_sr_store_promotion` | 外卖活动明细表 | 活动发布、配额管理、BD销单统计 |
| `dwd.dwd_sr_order_promotion_order` | 订单明细表 | 订单转化、AI外呼、实验归因 |

共设计了 **6大类、18项** 监控检查，按优先级 P0/P1/P2 分级：

|------|---------|--------|--------|--------|
| 1 | 数据量监控 | 日增量环比波动 | P0 | 全部 |
| 2 | 数据量监控 | 近7日趋势异常 | P1 | 全部 |
| 3 | 完整性 | 关键字段空值率 | P0 | 全部 |
| 4 | 完整性 | 非关键字段空值率 | P2 | 全部 |
| 5 | 唯一性 | 主键重复检查 | P0 | 全部 |
| 6 | 时效性 | 分区数据延迟 | P0 | 全部 |
| 7 | 一致性 | 用户ID关联校验 | P1 | 订单表 ↔ 用户表 |
| 8 | 一致性 | 活动ID关联校验 | P1 | 订单表 ↔ 活动表 |
| 9 | 有效性 | 订单状态分布异常 | P1 | 订单表 |
| 10 | 有效性 | 活动状态分布异常 | P1 | 活动表 |
| 11 | 有效性 | 支付金额异常值 | P2 | 订单表 |
| 12 | 有效性 | 注册时间异常值 | P2 | 用户表 |

## 二、各监控项详细设计

### 监控1：日增量环比波动 (P0)

**目的**：检测每日新增数据量的异常波动，是数据质量的第一道防线。

> **DDL 校正**：`dwd_sr_store_promotion` 的 dt 分区为**月级**（`date_trunc('month', dt)`），但 WHERE dt 条件仍按天过滤正常工作，仅分区裁剪为月级。

#### 1a. dim.dim_silkworm_user — 每日新增用户量

```sql
SELECT
    register_date AS `注册日期`,
    new_user_cnt AS `新增用户数`,
    LAG(new_user_cnt, 1) OVER (ORDER BY register_date) AS `前日新增用户数`,
    ROUND((new_user_cnt / LAG(new_user_cnt, 1) OVER (ORDER BY register_date) - 1) * 100, 2) AS `日环比变化率(%)`,
    LAG(new_user_cnt, 7) OVER (ORDER BY register_date) AS `7天前新增用户数`,
    ROUND((new_user_cnt / LAG(new_user_cnt, 7) OVER (ORDER BY register_date) - 1) * 100, 2) AS `周环比变化率(%)`,
    CASE
        WHEN ABS(new_user_cnt / NULLIF(LAG(new_user_cnt, 1) OVER (ORDER BY register_date), 0) - 1) > 0.30 THEN 'P1-危险'
        WHEN ABS(new_user_cnt / NULLIF(LAG(new_user_cnt, 1) OVER (ORDER BY register_date), 0) - 1) > 0.15 THEN 'P2-严重'
        ELSE '正常'
    END AS `告警级别`
FROM (
    SELECT
        substr(register_time, 1, 10) AS register_date,
        COUNT(DISTINCT user_id) AS new_user_cnt
    FROM dim.dim_silkworm_user
    WHERE substr(register_time, 1, 10) >= DATE_SUB(CURRENT_DATE(), INTERVAL 30 DAY)
    GROUP BY 1
) t
ORDER BY register_date DESC;
```

**告警阈值**：日环比 > ±30% (P1)，日环比 > ±15% (P2)

#### 1b. dwd.dwd_sr_store_promotion — 每日新增活动量

```sql
SELECT
    dt AS `日期`,
    promotion_cnt AS `新增活动量`,
    LAG(promotion_cnt, 1) OVER (ORDER BY dt) AS `前日新增活动量`,
    ROUND((promotion_cnt / NULLIF(LAG(promotion_cnt, 1) OVER (ORDER BY dt), 0) - 1) * 100, 2) AS `日环比变化率(%)`,
    LAG(promotion_cnt, 7) OVER (ORDER BY dt) AS `7天前新增活动量`,
    ROUND((promotion_cnt / NULLIF(LAG(promotion_cnt, 7) OVER (ORDER BY dt), 0) - 1) * 100, 2) AS `周环比变化率(%)`,
    CASE
        WHEN ABS(promotion_cnt / NULLIF(LAG(promotion_cnt, 1) OVER (ORDER BY dt), 0) - 1) > 0.30 THEN 'P1-危险'
        WHEN ABS(promotion_cnt / NULLIF(LAG(promotion_cnt, 1) OVER (ORDER BY dt), 0) - 1) > 0.15 THEN 'P2-严重'
        ELSE '正常'
    END AS `告警级别`
FROM (
    SELECT dt, COUNT(DISTINCT store_promotion_id) AS promotion_cnt
    FROM dwd.dwd_sr_store_promotion
    WHERE dt >= DATE_SUB(CURRENT_DATE(), INTERVAL 30 DAY)
    GROUP BY dt
) t
ORDER BY dt DESC;
```

#### 1c. dwd.dwd_sr_order_promotion_order — 每日订单量

```sql
SELECT
    dt AS `日期`,
    order_cnt AS `订单量`,
    valid_order_cnt AS `有效订单量`,
    cancel_order_cnt AS `取消订单量`,
    ROUND(valid_order_cnt / NULLIF(order_cnt, 0) * 100, 2) AS `有效率(%)`,
    -- 日环比
    ROUND((order_cnt / NULLIF(LAG(order_cnt, 1) OVER (ORDER BY dt), 0) - 1) * 100, 2) AS `日环比变化率(%)`,
    CASE
        WHEN ABS(order_cnt / NULLIF(LAG(order_cnt, 1) OVER (ORDER BY dt), 0) - 1) > 0.30 THEN 'P1-危险'
        WHEN ABS(order_cnt / NULLIF(LAG(order_cnt, 1) OVER (ORDER BY dt), 0) - 1) > 0.15 THEN 'P2-严重'
        ELSE '正常'
    END AS `告警级别`
FROM (
    SELECT
        dt,
        COUNT(1) AS order_cnt,
        COUNT(DISTINCT CASE WHEN order_status IN (2, 8) THEN order_id END) AS valid_order_cnt,
        COUNT(DISTINCT CASE WHEN order_status IN (4, 5) THEN order_id END) AS cancel_order_cnt
    FROM dwd.dwd_sr_order_promotion_order
    WHERE dt >= DATE_SUB(CURRENT_DATE(), INTERVAL 30 DAY)
    GROUP BY dt
) t
ORDER BY dt DESC;
```

### 监控2：关键字段空值率 (P0)

**目的**：检测核心字段是否出现大量NULL，反映上游数据是否完整。

#### 2a. dim.dim_silkworm_user 关键字段空值

```sql
SELECT
    CURRENT_DATE() AS `检查日期`,
    COUNT(1) AS `总用户数`,
    ROUND(SUM(CASE WHEN user_id IS NULL THEN 1 ELSE 0 END) / COUNT(1) * 100, 4) AS `user_id空值率(%)`,
    ROUND(SUM(CASE WHEN register_time IS NULL THEN 1 ELSE 0 END) / COUNT(1) * 100, 4) AS `register_time空值率(%)`,
    ROUND(SUM(CASE WHEN city_id IS NULL THEN 1 ELSE 0 END) / COUNT(1) * 100, 4) AS `city_id空值率(%)`,
    ROUND(SUM(CASE WHEN county_id IS NULL THEN 1 ELSE 0 END) / COUNT(1) * 100, 4) AS `county_id空值率(%)`,
    ROUND(SUM(CASE WHEN phone IS NULL OR phone = '' THEN 1 ELSE 0 END) / COUNT(1) * 100, 4) AS `phone空值率(%)`,
    CASE
        WHEN SUM(CASE WHEN register_time IS NULL THEN 1 ELSE 0 END) > 0 THEN 'P1-危险'
        WHEN SUM(CASE WHEN city_id IS NULL OR county_id IS NULL THEN 1 ELSE 0 END) / COUNT(1) > 0.05 THEN 'P2-严重'
        ELSE '正常'
    END AS `告警级别`
FROM dim.dim_silkworm_user
WHERE is_logoff = 0;
```

**关键字段**：`user_id`, `register_time`, `city_id`, `county_id`, `phone`

> **DDL 校正**：dim_silkworm_user 没有 `city_name` 列，只有 `city_id`(int)、`county_id`(int)；`register_time` 是 varchar 类型，`user_id` 是 bigint(20)。

#### 2b. dwd.dwd_sr_store_promotion 关键字段空值

```sql
SELECT
    CURRENT_DATE() AS `检查日期`,
    dt AS `分区日期`,
    COUNT(1) AS `总活动数`,
    ROUND(SUM(CASE WHEN store_promotion_id IS NULL THEN 1 ELSE 0 END) / COUNT(1) * 100, 4) AS `store_promotion_id空值率(%)`,
    ROUND(SUM(CASE WHEN store_id IS NULL THEN 1 ELSE 0 END) / COUNT(1) * 100, 4) AS `store_id空值率(%)`,
    ROUND(SUM(CASE WHEN merchant_id IS NULL THEN 1 ELSE 0 END) / COUNT(1) * 100, 4) AS `merchant_id空值率(%)`,
    ROUND(SUM(CASE WHEN begin_date IS NULL OR begin_date = '' THEN 1 ELSE 0 END) / COUNT(1) * 100, 4) AS `begin_date空值率(%)`,
    ROUND(SUM(CASE WHEN status IS NULL THEN 1 ELSE 0 END) / COUNT(1) * 100, 4) AS `status空值率(%)`,
    CASE
        WHEN SUM(CASE WHEN store_promotion_id IS NULL OR store_id IS NULL THEN 1 ELSE 0 END) > 0 THEN 'P1-危险'
        ELSE '正常'
    END AS `告警级别`
FROM dwd.dwd_sr_store_promotion
WHERE dt >= DATE_SUB(CURRENT_DATE(), INTERVAL 1 DAY)
GROUP BY dt;
```

**关键字段**：`store_promotion_id`, `store_id`, `merchant_id`, `begin_date`, `status`

#### 2c. dwd.dwd_sr_order_promotion_order 关键字段空值

```sql
SELECT
    CURRENT_DATE() AS `检查日期`,
    dt AS `分区日期`,
    COUNT(1) AS `总订单数`,
    ROUND(SUM(CASE WHEN order_id IS NULL OR order_id = '' THEN 1 ELSE 0 END) / COUNT(1) * 100, 4) AS `order_id空值率(%)`,
    ROUND(SUM(CASE WHEN user_id IS NULL THEN 1 ELSE 0 END) / COUNT(1) * 100, 4) AS `user_id空值率(%)`,
    ROUND(SUM(CASE WHEN store_promotion_id IS NULL THEN 1 ELSE 0 END) / COUNT(1) * 100, 4) AS `store_promotion_id空值率(%)`,
    ROUND(SUM(CASE WHEN user_pay_amt IS NULL THEN 1 ELSE 0 END) / COUNT(1) * 100, 4) AS `用户支付金额空值率(%)`,
    ROUND(SUM(CASE WHEN order_status IS NULL THEN 1 ELSE 0 END) / COUNT(1) * 100, 4) AS `order_status空值率(%)`,
    CASE
        WHEN SUM(CASE WHEN order_id IS NULL OR order_id = '' OR user_id IS NULL THEN 1 ELSE 0 END) > 0 THEN 'P1-危险'
        ELSE '正常'
    END AS `告警级别`
FROM dwd.dwd_sr_order_promotion_order
WHERE dt >= DATE_SUB(CURRENT_DATE(), INTERVAL 1 DAY)
GROUP BY dt;
```

**关键字段**：`order_id`, `user_id`, `store_promotion_id`, `user_pay_amt`, `order_status`

> **DDL 校正**：订单表没有 `pay_amt` 字段，实际金额字段为 `user_pay_amt`(用户支付金额) 和 `order_amt`(订单金额)。**order_id 在 DDL 中可为 NULL**，一旦出现空值即触发 P1 告警——这在业务上是严重的数据质量问题。

**告警阈值**：P0字段(主键/关联键)任何空值 → P1告警；非关键字段 > 5% → P2告警

### 监控4：分区数据延迟检测 (P0)

**目的**：检测数据是否按时产出，避免下游报表数据缺失。

```sql

SELECT
    CURRENT_DATE() AS `检查日期`,
    t.table_name AS `表名`,
    t.expected_dt AS `预期分区`,
    IFNULL(a.actual_cnt, 0) AS `实际数据量`,
    CASE
        WHEN IFNULL(a.actual_cnt, 0) = 0 THEN 'P1-数据缺失'
        WHEN IFNULL(a.actual_cnt, 0) < t.expected_min_cnt THEN 'P2-数据偏少'
        ELSE '正常'
    END AS `告警级别`
FROM (
    SELECT 'dwd.dwd_sr_store_promotion' AS table_name, DATE_SUB(CURRENT_DATE(), INTERVAL 1 DAY) AS expected_dt, 100 AS expected_min_cnt
    UNION ALL
    SELECT 'dwd.dwd_sr_order_promotion_order', DATE_SUB(CURRENT_DATE(), INTERVAL 1 DAY), 1000
) t
LEFT JOIN (
    SELECT 'dwd.dwd_sr_store_promotion' AS table_name, dt, COUNT(1) AS actual_cnt
    FROM dwd.dwd_sr_store_promotion
    WHERE dt = DATE_SUB(CURRENT_DATE(), INTERVAL 1 DAY)
    GROUP BY dt
    UNION ALL
    SELECT 'dwd.dwd_sr_order_promotion_order', dt, COUNT(1)
    FROM dwd.dwd_sr_order_promotion_order
    WHERE dt = DATE_SUB(CURRENT_DATE(), INTERVAL 1 DAY)
    GROUP BY dt
) a ON t.table_name = a.table_name AND t.expected_dt = a.dt;
```

### 监控5：用户ID关联校验 (P1)

```sql
    FROM dwd.dwd_sr_order_promotion_order
```

```sql
LEFT JOIN (
    SELECT store_promotion_id, COUNT(DISTINCT order_id) AS meituan_order_cnt
    FROM dwd.dwd_sr_order_promotion_order
    WHERE dt >= DATE_SUB(CURRENT_DATE(), INTERVAL 7 DAY)
      AND order_status IN (2, 8)
    GROUP BY store_promotion_id
) o ON p.store_promotion_id = o.store_promotion_id
WHERE p.status = 1  -- 1=已审核(有效活动)
  AND p.meituan_finished_num > 0;
```

## 三、帆软BI报表设计建议

### 报表结构（建议3个Tab页）

**Tab 1：监控总览仪表盘**
- 卡片：今日数据总量（3张表各一个卡片）

- 完整性：关键字段空值率柱状图（横向对比各字段）
- 一致性：关联缺失率趋势折线图

## 四、告警分级与通知策略

| 告警级别 | 定义 | 通知方式 | 响应时效 |
|---------|------|---------|---------|
| **P0 - 危险** | 数据缺失/主键重复/关键字段全空 | 钉钉群 @数据负责人 + 帆软BI推送 | 1小时内 |
| **P1 - 严重** | 关联断裂/业务逻辑错误/大幅波动 | 钉钉群通知 + 帆软BI推送 | 当日处理 |
| **P2 - 关注** | 轻度异常/异常值/趋势偏离 | 帆软BI报表标记 | 周报汇总 |

## 五、实施建议

### 分阶段实施

| 阶段 | 内容 | 预计工作量 |
|------|------|-----------|
| **第一阶段（MVP）** | 监控1(数据量)、监控2(完整性)、监控3(唯一性)、监控4(时效性) | 1天 |
| **第二阶段** | 监控5(用户关联)、监控6(活动关联)、监控7(订单状态)、监控8(活动状态) | 1天 |
| **第三阶段** | 监控9-16(业务逻辑+交叉校验) | 1天 |
| **第四阶段** | 帆软BI报表搭建 + 告警配置 + 试运行调参 | 2天 |

### 阈值调优说明
- 初始阈值基于经验设定，建议试运行1-2周后根据实际数据波动调整
- 活动量/订单量波动受业务节奏影响大（周末效应、节假日、活动大促），建议剔除已知业务事件后再判定告警

## 七、待确认事项

1. **数据产出时间**：各表 T 日数据预计几点前产出？决定调度时间。
2. **告警接收人**：钉钉群和帆软BI推送到哪些人？
3. **order_id 去重策略**：既然 order_id 可为 NULL 且主键是 auto_id，是否需要按 order_id 做业务去重（排除 NULL）？

5. **是否已有部分监控在运行**？避免重复建设。
6. **varchar 时间字段格式**：order_time, auth_order_finish_time 等 varchar 字段的存储格式是否统一为 `YYYY-MM-DD HH:MM:SS`？影响时间比较类监控的可靠性。
