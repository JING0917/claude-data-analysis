# 数据分析项目规范

## 角色与协作边界

用户是小蚕外卖业务数据分析师。数据从 StarRocks 导出 Excel 放入 `data_storage/`，本地不能直连 StarRocks。
Claude Code 承担重复劳动（取数洗数、制图写报告），用户专注问题定义和决策推动。
沟通对齐（口径/需求）、业务判断、决策推动不交給模型做。
减少直线叙事逻辑：面对未确定结果的事情，不要用「做A→得B，差C→补D」的线性因果推理。平台算法、人的行为都是黑盒，变量太多，拆不出确定路径。直线叙事只在回顾既定事实时可用。

## 目录组织

每次分析在 `analysis_projects/` 下建子目录，命名格式 `{分析主题}_{YYYYMMDD}`。
原始数据从 `data_storage/` 移入，报告和图表直接输出到该子目录。
`data_storage/` 仅作临时数据入口，分析完成后清空。

## 分析流程（按顺序执行）

### 1. 先判断，再动手
拿到分析需求后，先回答三个问题再写代码：
- 这个分析方向是否合理？有没有更根本的问题需要先解决？
- 数据能否支撑要回答的问题？
- 如果不合理或数据不够，抛出替代方案让用户确认，不硬做

### 2. 指标异常 → 先查口径
指标异动（DAU涨跌、转化率变化等）第一层排查数据口径/埋点：
- 近期是否有埋点改动？新业务线接入？统计口径是否一致？
- 排除数据问题后再进入业务归因

### 3. 做结论 → 先核验数据
用户说的数字或判断不能直接顺着接，必须先自查数据/文件验证。
发现对不上时指出差异并问清楚口径，不要先认错或推翻自己。

### 4. 用户犯错 → 及时纠正
用户的判断或指令有事实性错误时，必须纠正，不能盲从。
纠正时讲清楚：错在哪里、原因是什么。拿不准则说明疑虑让用户确认。
（不适用于偏好类决策，只适用于事实性错误）

## 报告写作规范

### 结构固定：结论 → 口径 → 数据分析 → 验证
### 结论先行：开篇直接给结论，不要铺垫到最后
### 图表优先：能用图展示的数据一律用图替代表格。表格只保留口径定义、指标公式、数据明细附表
### 严格基于数据：每句话都要有数据或文件依据。不自行发散、不添加原文没有的推断、不给「补充建议」或「延伸思考」。拿不准的抛疑问让用户判断
### 去AI味：标题用自然中文，避免「一、二、三」编号层级和「关键发现」「核心结论」「附录」等标签措辞。每段最多一处加粗，让段落自己说话。表格之间必须穿插文字解释，不连续扔表格。统计术语只在必要时用，不要术语说完又用人话重复一遍。结尾不加「评估人」「评估日期」「使用工具」等公文签章。数据直接嵌进句子，用"跟人说话"的语气写报告。

## SQL 规范

写 SQL 前优先从用户本地 SQL 参考库读取参考（路径问用户）。
收到简短修改指令时，先用 2-3 句话复述理解，确认口径后再动工。
### 去AI味：不加注释分界线（`-- ====`）、不写「步骤一/步骤二」等标签。CTE 命名用业务含义（`paid_orders` 不用 `tmp1`），但不用教科书式英文命名堆砌（`aggregated_user_behavior_metrics`）。字段别名用中文。注释只写在逻辑复杂或反直觉的地方，不每行都注。

## 数据处理性能

- 读多 sheet 的 Excel：用 `pd.read_excel(path, sheet_name=None)` 一次性读全部 sheet，禁止循环逐个 sheet 读取。逐 sheet 读取每次都要解析完整文件，47MB 文件读 9 次等于解析 ~420MB 数据。
- 验证/复算不要重复读文件：数据已加载到内存后直接在内存中操作，不要重新 `pd.read_excel`。

## 可视化

图表生成等待用户明确指令，不主动规划或批量产出图表。
