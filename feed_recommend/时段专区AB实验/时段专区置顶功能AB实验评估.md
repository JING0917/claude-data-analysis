# 时段专区活动置顶功能 — AB实验评估

> 评估日期：2026-05-12 | 实验上线：2026-04-16

---

## 一、结论

**前后对比不可归因（有外部因素混淆），但分组对比可以做。建议直接用匹配到的用户拉实验组 vs 对照组 CTR 对比，即可自证置顶效果。**

---

## 二、实验背景

| 项目 | 内容 |
|------|------|
| 实验功能 | 用户从首页 feed 运营位点击活动 → 进入时段专区后，该活动在列表页置顶展示 |
| 实验组 | 有置顶功能（50% 用户） |
| 对照组 | 无置顶功能（50% 用户） |
| 分流触发条件 | 用户点击首页 feed 运营位 → 进入时段专区 |
| 入口情况 | **时段专区列表页仅此一个入口**（首页 feed 运营位） |
| 上线时间 | 2026-04-16 晚间 |

---

## 三、前后对比 T 检验（不分实验组/对照组）

对比 4/6-12（实验前）vs 4/20-26（实验中），双侧同方差 T 检验。各指标均为**先按日计算、再取周日均值**。

### 指标口径

| 指标 | 计算方式 | 说明 |
|------|------|------|
| DAU | 每日活跃用户数，取周日均 | 日活用户去重 |
| 时段专区活动CTR(次数) | 时段专区活动点击次数 / 活动曝光次数，每日一个值，取周日均 | 事件粒度，`event = Recommend_Activity_Ex` |
| 时段专区活动CTR(用户) | 时段专区活动点击UV / 活动曝光UV，每日一个值，取周日均 | 用户粒度 |
| DAU→时段专区活动曝光转化 | 时段专区活动曝光UV / DAU，每日一个值，取周日均 | |
| DAU→时段专区活动点击转化 | 时段专区活动点击UV / DAU，每日一个值，取周日均 | |

### 结果

| 指标 | 实验前(4/6-12) | 实验中(4/20-26) | 变化 | P值 |
|------|:--:|:--:|:--:|:--:|
| DAU（周日均） | 103.8万 | 110.3万 | **+6.2%** | <0.001 |
| 时段专区活动CTR(次数) | 4.08% | 4.55% | +11.6% | <0.001 |
| 时段专区活动CTR(用户) | 66.66% | 72.53% | +8.8% | <0.001 |
| DAU→活动曝光转化 | 1.61% | 2.30% | +42.8% | <0.001 |
| DAU→活动点击转化 | 1.07% | 1.67% | +55.5% | <0.001 |

所有指标显著提升，方向一致。但 DAU 同期增长了 6.2%——置顶功能不可能拉动 DAU，说明**实验期间存在外部利好**（季节变化、营销活动等），这些因素会同时推高时段专区指标。前后对比不能归因于置顶功能。

## 四、分组对比 — 为什么不致命

### 4.1 时段专区只有一个入口

时段专区列表页**只有首页 feed 运营位这一个入口**。所有进入时段专区的用户都经过了实验分流点，理论上 100% 应被分配到实验组或对照组。

### 4.2 匹配率低是数据链路问题，不是覆盖问题

SQL 跑出来的匹配率：

| 日期 | 时段专区活动曝光UV | 能匹配到分流记录的用户 | 匹配率 |
|------|:--:|:--:|:--:|
| 4/17 | 18,000+ | 2,400+ | ~13% |
| 4/18 | 19,000+ | 1,600+ | ~8% |
| 4/19 | 20,000+ | 1,000+ | ~5% |

之前误以为是「其他入口流入」，实际原因是**后端分流记录只保留最新一条**，历史分流数据被覆盖导致匹配断裂。这是数据链路 bug，不是实验设计问题。

**丢记录是随机丢失**（跟用户特征无关），匹配到的用户仍是有效的随机样本。匹配率低会降低统计检验力，但不会让结论失效。

### 4.3 逻辑链

1. 时段专区只有一个入口 → 所有用户都是目标人群
2. 后端丢记录是随机的 → 匹配到的用户仍是有效随机样本
3. 实验组 vs 对照组 CTR 对比 → 直接自证

**下一步**：直接用匹配到的用户（实验组 vs 对照组）跑时段专区活动 CTR 对比，即可得出结论。

---

## 五、补充：这个功能真的需要 AB 吗

置顶不是「策略优化」，是**体验修复**。用户在 feed 运营位点击了一个活动，跳转到时段专区后要在一堆活动里重新找到刚才那个——这个体验本身就是断裂的。置顶只是让用户看见自己已经选中的东西。等同于：点了搜索结果，跳转后不用重新找。

如果非要一组数据自证，实验组 vs 对照组 CTR 对比跑一下就够了，不需要复杂的评估框架。

## 六、后续建议

| 要做的事 | 说明 |
|------|------|
| **先跑分组对比** | 用已有匹配数据（实验组 vs 对照组）拉 CTR，方向 + 量级一目了然 |
| 修复后端分流记录 | 改为全量保留，避免历史记录被覆盖 |
| 长期：客户端上报 | 实验分流埋点从后端记录改为客户端上报，从根源解决匹配断裂 |
